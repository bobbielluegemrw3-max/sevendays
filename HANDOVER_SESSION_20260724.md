## 1. ★現在の作業モード＝「レビュー側→開発側の1ページずつ改修」（明記）

このプロジェクトは今、**1ページずつ順番に品質を上げていく改修フェーズ**にある。1ページごとに次のループを回す:

```
① レビュー側(Claude)が対象1ページを監査し、改修仕様MDをリポジトリルートに置く
      （例: RACE_PAGE_BELOW_MONITOR_SPEC.md / HORSE_PAGE_*_BRIEF.md）
② オーナーが方針を決める（例: 道A / A案 / 忠実写経）
③ デザイン側が必要なら視覚をZIPハンドオフ（.dc.html + 画像 + 仕様）
④ 開発側(この端末)が実装 → 必ず /dev/*-preview にまず当てる（本番直出し禁止）
⑤ オーナーが実機スクショで確認 → 修正ループ → 承認
⑥ 承認後に本番結線 → /healthz でSHA一致確認
⑦ 開発側が完了報告MDをルートに置く（実装側→レビュー側）→ レビュー側が突合
```

**4者の分業（前任から継続）**:
- **レビュー側** = 情報設計・仕様判断（※事実誤認するので、指摘は必ずコードで実在確認してから当てる）
- **デザイン側** = 視覚（ZIP handoff・.dc.html/canvas も含む）。※依頼範囲を超えて「元の美しい盤」まで作り変えることがある → **元UI厳守を明示せよ**
- **オーナー** = 実機での最終判断（**スクショが撮れるのはオーナーだけ**）
- **開発側(実装)** = コード検証・実装・記録（**この端末は視覚を撮れない**＝プレビュー＋オーナースクショが命綱）

**完了したページ**: ①厩舎ページ ②馬個別ページ ③レースページ。**次のページはオーナー／レビュー側が次の仕様MDで指定する**（§6）。

## 4. ★鉄則・教訓（前任から継続＋本セッションで再確認）

1. **見た目/レイアウトは本番直出し禁止。** 必ず `/dev/*-preview` で作り、オーナーのスクショ→修正ループで固めてから本番結線（前任の馬個別ページ大破の教訓）。本セッションは全ページこの流れを守り事故ゼロ。
2. **デザイン側ハンドオフは「忠実写経」。** 自己流で仕上げない。分からなければ推測せずオーナー/デザイナーに聞く（C/D却下の教訓）。gzip梱包 Design Component は python `json.loads` でデコード→正典HTMLを取り出して1対1で写経。
3. **「元の美しい盤」は変えない。** デザイン側は依頼範囲を超えて既存UIまで作り変えることがある。演出は**上に薄く重ねるだけ**（`ReachFxLayer` は overlay・元の `BandRaceAct` 不変）。
4. **レビュー指摘は鵜呑みにせず実在確認。** 「ダブルスタンダード」指摘が実は過去の意図的決定だった例あり。過去の Decision / ブリーフをコードで確認してから動く。
5. **tsc が通っても Next（Turbopack）ビルドは別に落ちる。** CSS Modules 純粋セレクタ必須。フルゲート（typecheck+lint+build+test）必須。
6. **push後は `/healthz` の `build` フィールドでSHA一致を確認**してからオーナーに見せる（`commit` ではなく `build`）。
